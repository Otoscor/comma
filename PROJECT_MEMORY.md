## Quick Start (새 에이전트를 위한 빠른 시작)

### 프로젝트 정의
**comma**는 웹소설 독자가 텍스트를 AI로 시각화하고, 다른 독자들의 해석을 소셜 플랫폼에서 공유·발견할 수 있는 차세대 독서 플랫폼입니다.

### 핵심 컨셉 3가지
1. **텍스트 스크랩 기반 AI 생성**: 독자가 원하는 장면을 드래그하면 AI가 이미지/비디오/음성 생성
2. **로그(Log)**: 모든 독자의 시각화가 모이는 소셜 공유 공간
3. **마커 시스템**: 본문 위에 다른 독자들의 시각화 위치를 원형 마커로 표시

### 필수 읽기 문서
1. **README.md** - 프로젝트 개요 및 문서 인덱스
2. **PROJECT_MEMORY.md** (이 문서) - 작업 컨텍스트 및 히스토리
3. **SERVICE_PLAN.md** - 서비스 기획 상세 (v0.2)

## 핵심 개념 (Key Concepts)

### 로그 (Log)
**정의**: 해당 웹소설에 대한 모든 독자들의 시각화 콘텐츠가 모이는 공유 공간

**작동 방식**:
1. 내가 문단을 시각화하면 → 해당 소설의 로그에 자동 push
2. "로그 보기" 활성화 → 본문 위에 다른 독자들이 시각화한 위치를 **마커**로 표시
3. 마커 클릭 → 해당 독자가 생성한 이미지/비디오/음성 감상
4. 좋아요, 저장, 댓글 등 소셜 반응 가능

**특징**:
- 단순 저장소가 아닌 **소셜 발견 플랫폼**
- 같은 장면에 대한 **다양한 해석** 비교 가능
- 마커 밀도로 **인기 장면** 자연스럽게 발견

### 마커 (Marker)
**정의**: 본문 텍스트 위에 표시되는 원형 인디케이터로, 다른 독자가 해당 위치를 시각화했음을 나타냄

**기능**:
- 독자들의 활동을 히트맵처럼 시각화
- 클릭 시 해당 독자의 로그(시각화 콘텐츠) 표시
- 마커가 많을수록 인기 있는 장면

### AI 생성 플로우
**텍스트 → 이미지/비디오/음성**

**프로세스**:
1. 독서 중 텍스트 드래그/선택
2. 플로팅 툴바에서 생성 타입 선택 (이미지/비디오/음성/메모)
3. 바텀시트에서 옵션 확인 및 "만들기"
4. AI 생성 (로딩)
5. 결과 확인 → "로그에 넣기" 또는 "다시 만들기"
6. 로그에 자동 저장 → 본문에 내 마커 추가

**특징**:
- 읽기 흐름을 끊지 않는 seamless UX
- 소설 전체 맥락 기반 생성 (단순 프롬프트 아님)
- 독자가 원하는 장면을 선택 (작가가 아닌)

## 주요 결정사항 (Key Decisions)

### 1. 로그를 소셜 플랫폼으로 정의
**결정**: 로그를 단순 개인 저장소가 아닌 소셜 공유 공간으로 설계

**이유**:
- 같은 소설을 읽는 독자들 간 커뮤니티 형성
- 다양한 해석 발견 및 영감 제공
- 독자 참여도 증가 (좋아요, 댓글 등)

**영향**:
- 마커 시스템 도입
- 로그 갤러리 UI 필요
- 소셜 기능 (좋아요, 댓글) 추가

### 2. 마커 오버레이 시스템
**결정**: 본문 텍스트 위에 다른 독자들의 활동을 마커로 표시

**이유**:
- 인기 장면을 자연스럽게 발견
- 다른 독자들의 시각화 위치 한눈에 파악
- 텍스트 읽기와 소셜 기능의 자연스러운 통합

**기술 고려사항**:
- 텍스트 오프셋 기반 마커 위치 계산 필요
- 마커 클러스터링 (같은 위치에 여러 마커)
- 마커 표시/숨김 토글 기능

### 3. PascalCase 네이밍 컨벤션
**결정**: Figma 프레임/컴포넌트에 PascalCase 네이밍 채택

**이유**:
- Figma 권장 패턴
- 개발 시 코드로 직접 변환 가능
- 계층 구조 표현 (점 구분자 사용)

**적용 범위**:
- 화면: `ReaderScreen_Default`
- 컴포넌트: `Navigation.AppBar`
- 레이어: `Text_NovelTitle`

### 4. 3-Phase 화면 구조
**결정**: 9개 프레임을 3개 Phase로 그룹화

**Phase 구분**:
- **Phase 1**: 웹소설 읽기 (Frame 1-3)
- **Phase 2**: AI 콘텐츠 생성 (Frame 4-8)
- **Phase 3**: 로그 갤러리 (Frame 9)

**이유**:
- 개발 우선순위 명확화 (MVP 범위 설정)
- 각 Phase별 독립적 개발 가능
- 기능 단위 테스트 및 배포

### 5. 점(.) 구분자로 계층 표현
**결정**: 컴포넌트 계층 구조를 슬래시(/)가 아닌 점(.)으로 표현

**변경 전**: `Navigation/AppBar`, `Button/Primary`
**변경 후**: `Navigation.AppBar`, `Button.Primary`

**이유**:
- 개발 친화적: 많은 프로그래밍 언어의 네임스페이스 패턴과 일치
- React/Vue 등에서 `Navigation.AppBar`로 직접 참조 가능
- 파일 시스템 경로와 혼동 방지 (슬래시는 경로로 인식됨)

**영향**:
- NAMING_CONVENTION.md v1.1로 업데이트
- RENAMING_QUICK_REFERENCE.md 전체 예시 업데이트
- 모든 컴포넌트 네이밍 패턴 변경
